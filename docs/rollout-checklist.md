# ERP Tiered Rollout + Regression Checklist

이 문서는 ERP 쓰기 액션의 중복 클릭/중복 요청 방지를 **Tier A -> Tier B -> Tier C** 순서로 안전하게 배포하기 위한 운영 체크리스트입니다.

핵심 원칙:

- 기존 ERP 상태 전이/업무 규칙은 변경하지 않고, 가드/멱등성 레이어를 덧씌운다.
- 단계별 적용과 회귀 검증을 통과해야 다음 Tier로 확장한다.
- 기존 프로덕션 플로우와 다른 동작이 관측되면 즉시 확장 중단 후 원인 분석한다.

## 1) 비즈니스 플로우 보호 게이트 (모든 Tier 공통)

배포 전/후 아래 항목을 반드시 확인:

- [ ] 기존 상태값/상태 전이 규칙(승인, 출고, 재고, 문서)의 명칭/순서/조건 변경 없음
- [ ] 기존 RPC 시그니처와 UI 호출 계약 유지 (호출 파라미터/응답 형태 호환)
- [ ] 승인 라인 순서 의미(결재 순차성) 변경 없음
- [ ] 재고 원장 방향/부호/수량 계산 규칙 변경 없음
- [ ] 신규 제약(유니크/인덱스/체크)은 기존 데이터 백필 검토 후 적용
- [ ] 동일 요청 재시도 시 결과는 `replayed`/`no_op`로 일관되고 부작용 없음

중단 조건 (Stop-the-line):

- [ ] 기존 정상 시나리오(예: draft -> submit -> approve/reject)가 실패하거나 분기 동작이 달라짐
- [ ] 동일 idempotency 키에서 payload 충돌 검증이 누락되어 중복 쓰기 발생 가능
- [ ] 트랜잭션 경계 밖에서 부분 쓰기(partial write) 가능성 확인됨

## 2) 공통 응답/상태 계약

모든 쓰기 엔드포인트/서버 액션은 아래 상태 계약을 사용:

- `processed`: 첫 정상 처리
- `replayed`: 동일 요청 재시도, 기존 결과 재응답
- `no_op`: 이미 목표 상태라 추가 쓰기 없음
- `rejected`: 상태 충돌/검증 오류/권한 오류/서버 오류

권장 HTTP 매핑:

- `200`/`201`: `processed`, `replayed`, `no_op`
- `409`: `rejected` + `invalid_pre_state`/`conflict`
- `400`: `rejected` + `validation`
- `401`/`403`: `rejected` + `permission`
- `500`: `rejected` + `server_error`

## 3) Tier A 체크리스트 (고위험: 재고/승인/비가역 전이)

대상 예시:

- 재고 이동/조정, 승인 최종 처리, 출고 확정/취소 등 상태 전이 불가역 또는 재고 영향 액션

구현 체크:

- [ ] 클라이언트 단일 제출 가드 적용 (첫 클릭 즉시 lock, 핸들러 early return, 일관된 disabled/loading)
- [ ] 서버 선행 상태 조건부 업데이트 적용 (유효 pre-state에서만 전이 허용)
- [ ] idempotency 키 저장 및 재시도 재응답(`replayed`/`no_op`) 적용
- [ ] 동일 키 + 상이 payload는 `409 conflict`로 강제 차단
- [ ] 다중 쓰기(헤더/라인/원장)는 단일 트랜잭션으로 원자성 보장

회귀 테스트 (Tier A 통과 조건):

- [ ] 첫 요청 `processed`
- [ ] 즉시 재시도 `replayed` 또는 `no_op`
- [ ] 동일 키 + 다른 payload `409 conflict`
- [ ] 잘못된 pre-state `409 invalid_pre_state` + 부작용 없음
- [ ] 권한 부족 시 `403 rejected(permission)` + 데이터 미변경
- [ ] 핵심 여정 무회귀: draft -> submit -> approve/reject
- [ ] 핵심 여정 무회귀: outbound request submit/cancel
- [ ] 핵심 여정 무회귀: inventory increment/decrement write path

배포 게이트:

- [ ] Tier A 회귀 테스트 100% 통과
- [ ] 운영 모니터링에서 중복 쓰기/부분 쓰기 징후 없음
- [ ] 도메인 오너 승인 후 Tier B 진행

## 4) Tier B 체크리스트 (중위험: 문서 생성/수정/삭제)

대상 예시:

- 구매/생산/견적/출고요청/고객/품목 등 문서성 write

구현 체크:

- [ ] Tier A에서 확정한 단일 제출 패턴을 동일하게 적용
- [ ] create 계열은 중복 민감 필드 기준 dedupe 전략 정의 (idempotency 키 또는 유니크 기반)
- [ ] update/delete 계열은 stale 상태 방지 가드(버전/상태 검증) 적용
- [ ] 실패 응답이 기존 화면 UX(확인창/리다이렉트/토스트)와 충돌하지 않는지 확인

회귀 테스트 (Tier B 통과 조건):

- [ ] 빠른 연속 클릭에서도 문서 1건만 생성/수정/삭제
- [ ] 브라우저 재시도/네트워크 재전송에서도 중복 레코드 미생성
- [ ] 기존 문서 라이프사이클(작성 -> 수정 -> 확정/취소) 무회귀
- [ ] 권한/검증 실패 시 기존 메시지/동작 규칙 유지

배포 게이트:

- [ ] Tier B 모듈별 체크리스트 완료
- [ ] Tier A 지표 악화 없음 확인
- [ ] 도메인 오너 승인 후 Tier C 진행

## 5) Tier C 체크리스트 (저위험: 게시판/댓글 등)

대상 예시:

- 그룹웨어 게시판/댓글/첨부 등 상대적 저영향 write

구현 체크:

- [ ] 공통 단일 제출 패턴 적용
- [ ] 중복 생성 방지(키 또는 짧은 시간 dedupe) 적용
- [ ] 실패/재시도 시 사용자 피드백 문구 일관성 유지

회귀 테스트 (Tier C 통과 조건):

- [ ] 더블클릭 시 중복 게시글/댓글 미생성
- [ ] 첨부 업로드 재시도 시 동일 파일 중복 레코드 정책대로 동작
- [ ] 기존 저널/알림/피드 반영 규칙 무회귀

배포 게이트:

- [ ] Tier C 체크리스트 완료
- [ ] 전 Tier 운영 안정성 유지
- [ ] 전체 롤아웃 완료 승인

## 6) 운영 검증 및 롤백 준비

릴리스 직전:

- [ ] 모듈별 변경점/영향 범위/모니터링 포인트 문서화
- [ ] 장애 시 즉시 비활성화 가능한 feature scope 또는 대응 절차 준비

릴리스 직후:

- [ ] 중복 요청 비율, `conflict`, `invalid_pre_state`, `permission` 응답 추이 관찰
- [ ] 재고/승인/출고의 핵심 KPI 이상 징후 확인
- [ ] 이상 징후 발생 시 마지막 안정 범위(Tier)로 확장 중단 및 원복 판단
