# Approval / outbound — popup, 승인 철회, 삭제 manual checklist

Use when validating 전자결재 스pec work: popup 상세, 승인 철회, 기안자·관리자 삭제. Tick items as you go.

## Preconditions

- Two browser profiles or incognito + normal: **기안자**, **결재자**(순차 2인 이상이면 더 좋음), **시스템/결재 관리자** (`is_approval_admin`).
- Pop-up blocker **off** for the app origin (or verify 폴백 동작).

---

## 1. 문서번호 → 팝업 상세 (`/approvals/view/…`, `/outbound-requests/view/…`)

1. **통합 결재함** (`/approvals`): 문서번호 클릭 → 새 창/탭.
   - Expected: **사이드바 없음**(베어 셸), 본문·용지 **읽기 전용**; 결재 권한이 있으면 **하단 액션**(승인/반려 등) 유지.
2. **대시보드** 결재 대기 행: 동일하게 번호/링크로 팝업 열림.
3. **출고 목록**: 출고 연결 문서 번호 → `/outbound-requests/view/{id}` 동일 정책.
4. **창 크기**: 작성 팝업과 동일한 크기·옵션(예: 가로·세로 큰 팝업)인지 육안 비교.
5. **팝업 차단 시**: `window.open` 실패 시 **현재 탭 이동** 등 작성 화면과 같은 폴백.
6. **관리자 삭제·액션 후**: `opener`가 있으면 **목록/대시보드 새로고침**(`opener.location.reload`) 되는지 확인(크로스 오리진이면 생략될 수 있음).

---

## 2. 승인 철회 (직전 본인 승인 + 다음 차수 미처리)

1. 순차 결재선에서 **A가 승인**, **B가 아직 대기**인 문서로 **A 계정** 로그인.
   - Expected: **승인 철회** 버튼 노출(본인 `approved` 결재·협조·참조 액션 라인이고, 다음 액션 라인이 `pending`/`waiting`).
2. A가 아직 승인 전이거나, B가 이미 처리한 뒤 → 철회 버튼 **비노출**.
3. 철회 실행 후:
   - 문서 `status`·라인 상태가 스펙대로(예: 반려 종료 등)인지, **기안자**에게 **반려와 동등한 편집/삭제·재상신** UX가 적용되는지 확인.
4. **출고 요청** 동일 화면/컴포넌트를 쓰는 경우 동일 시나리오로 한 번 반복.

---

## 3. 삭제

### 3a 기안자 삭제 (`canWriterDeleteApprovalDoc`)

1. **반려** 종료 문서: 상세에서 삭제 가능, 삭제 후 목록에서 사라짐.
2. **회수**로 `draft` 복귀(`remarks`에 `기안 회수됨`): 삭제 가능.
3. **일반 임시저장**(`draft`, 회수 비고 없음): 삭제 **불가** UI.
4. **진행 중**(`in_review` 등): 기안자 삭제 **불가**.

### 3b 관리자 문서 삭제 (`admin_delete_approval_doc`)

1. 관리자 계정으로 상세(팝업 또는 풀 셸) → **문서 삭제**.
2. 확인창 문구: `현재 상태: {label}입니다. 그래도 삭제하시겠습니까?` 에서 `{label}`이 화면 **첫 상태 배지** 텍스트와 **일치**하는지 확인(단위 테스트 `lib/approval-status.test.ts`와 동일 기준).
3. 취소 → 문서 유지. 확인 → RPC 성공, 목록 갱신, 출고 문서면 **재고/연동** 스펙에 맞게 정리되는지 smoke.

---

## Pass criteria

- 팝업 경로·베어 셸·읽기 전용 본문이 **작성 팝업과 정책 일치**.
- 승인 철회는 **허용된 경우에만** 노출되고, 이후 상태·기안자 권한이 스펙과 맞음.
- 기안자 삭제는 **반려·회수 draft**만 허용; 관리자 삭제는 **확인 문구 라벨**이 상태 배지와 일치하고 RPC가 동작함.
